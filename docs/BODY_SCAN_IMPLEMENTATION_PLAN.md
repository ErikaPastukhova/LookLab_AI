# План улучшения скана тела по фото

## Контекст

На странице онлайн-манекена есть модуль "Скан тела" в `frontend/script.js`.
Сейчас он работает полностью в браузере:

1. Загружает MediaPipe Pose Landmarker.
2. Находит pose landmarks на фронтальном фото и опционально на фото сбоку.
3. Строит foreground mask примитивным отделением человека от фона по цвету.
4. Ищет горизонтальные сечения силуэта для груди, талии и бедер.
5. Переводит ширину сечений в обхваты через эллипс и эвристические коэффициенты.
6. Руку и ногу оценивает по длине костевых сегментов с антропометрическими множителями.

Текущие симптомы:

- часто "недостаточно уверенно" и результата нет;
- при ослаблении порога появляются странные числа;
- грудь нередко включает плечи/подмышки/руки;
- руки, волосы, фон, тени и одежда сильно искажают силуэт;
- бицепс и нога считаются как грубые догадки, а не измерения ширины.

Главная причина: confidence сейчас оценивает не анатомическую правильность мерки, а только геометрическую согласованность выбранного сечения с ожидаемым коридором между плечами и бедрами. Если выбранное сечение стабильно проходит по плечевому поясу, оно может выглядеть "уверенным", но всё равно быть неверной грудью.

## Цели

- Сделать скан предсказуемым: пользователь должен понимать, почему результат принят или отклонен.
- Убрать попадание плеч/рук в грудь, талия и бедра должны искаться в анатомически корректных зонах.
- Заменить цветовую маску фона на более надежную сегментацию человека.
- Перестать выдавать "точные" значения там, где алгоритм на самом деле делает статистическую догадку.
- Сохранить демо-ценность проекта: даже при неполном автоматическом расчете пользователь должен быстро получить редактируемые параметры.

## Не цели первой итерации

- Медицинская или швейная точность.
- Расчет реальных мерок по одному casual-фото без ограничений.
- Полная серверная CV-инфраструктура.
- Поддержка сложных поз, оверсайз-одежды, боковых ракурсов вместо фронта.

## Статус реализации на 2026-05-10

Реализована первая практическая итерация без подключения новой segmentation-модели.

Что сделано:

1. Расчет скана вынесен из большого `frontend/script.js` в тестируемый модуль `frontend/body-scan/bodyScanCore.js`.
2. Добавлены проверки core landmarks, наклона плеч/бедер, фронтальности и риска рук рядом с корпусом.
3. Окно груди перенесено ниже плечевого пояса; выбор груди больше не является простым максимумом ширины.
4. Результат теперь разделяет методы:
   - `measured` - фронт + профиль;
   - `frontApprox` - ширина с фронта + оценка глубины;
   - `maskSection` - сечение конечности по маске;
   - `lengthApprox` - приближение по пропорциям;
   - `statFallback` - статистическая оценка по росту/весу/полу.
5. Если торсовые сечения не найдены, скан больше не упирается в пустой результат: грудь/талия/бедра заполняются через `statFallback` с низкой уверенностью и явной причиной.
6. Для бицепса и ноги добавлены сечения перпендикулярно сегментам shoulder-elbow и hip-knee вместо прямой зависимости только от длины сегмента.
7. Для конечностей добавлен sanity clamp, чтобы UI не получал физически и интерфейсно странные значения.
8. В `?debugBodyScan=1` добавлены overlay окон поиска и структурированный debug-result в `window.__lastBodyScanResult`.
9. Подсказка в модалке изменена: руки нужно слегка отводить от корпуса, чтобы между руками и телом был виден фон.
10. Добавлен deploy-mapping для нового модуля `body-scan/bodyScanCore.js`.

Новые находки по локальному тестированию:

- На переданных фронтальных фото без бокового снимка MediaPipe часто находит позу, но цветовая маска торса не дает надежных горизонтальных сечений. В этих случаях правильнее показывать `statFallback`, а не пытаться расширять пороги и получать грудь вместе с плечами/руками.
- Фото с руками близко к корпусу почти всегда нужно считать низконадежными для груди и талии. Это не лечится одним confidence threshold.
- Headless/browser-тест выявил отдельную устойчивостную ошибку: если WebGL недоступен, `new THREE.WebGLRenderer()` валил весь `script.js`, и скан тела вообще не инициализировался. Добавлен fallback: 3D-превью отключается, но UI и скан продолжают работать.
- Полноценная точность по одному фронтальному фото невозможна без либо профильного фото, либо body-part/person segmentation. Текущая первая итерация делает результат редактируемым и честно помечает приблизительные значения.

Проверки, которые теперь есть:

- `scripts/test_body_scan_core.mjs` - unit-тесты post-processing на synthetic masks: окно груди ниже плеч, предупреждение о руках, отказ при отсутствующих landmarks, `statFallback` при провале маски торса.
- `scripts/body_scan_browser_smoke.mjs` - browser smoke-test через Chrome DevTools Protocol: открывает локальную страницу, загружает фото из `test_photo`, запускает расчет и печатает итоговые методы/значения/причины.

## Целевая логика

Идеальная схема:

1. Пользователь загружает фронтальное фото.
2. Пользователь опционально загружает профильное фото.
3. Алгоритм валидирует качество входа: полный рост, фронтальность, видимость ключевых точек, руки не сливаются с корпусом.
4. Алгоритм строит person segmentation mask.
5. Из маски вычитаются руки для расчета торса.
6. Грудь/талия/бедра ищутся в анатомических зонах, привязанных к landmarks и пропорциям тела.
7. Глубина берется из профильного фото; без него результат помечается как приблизительный.
8. Пользователь видит overlay линий, confidence и причины предупреждений.
9. Пользователь может поправить значения вручную перед применением к манекену.

Цель этапа: понять, где именно ломается текущий алгоритм на реальных фото, и перестать отлаживать "вслепую".

2. В debug mode рисовать поверх фото:
   - MediaPipe landmarks плеч, бедер, локтей, коленей;
   - текущие линии груди/талии/бедер;
   - границы foreground mask;
   - анатомические окна поиска груди/талии/бедер;
   - подписи confidence и source (`centerCrossing`, `consensus`, `longestFallback`, `statFallback`).

3. Вывести `qualityReasons` пользователю или в debug panel.
   Сейчас массивы причин формируются, но фактически не помогают пользователю понять проблему.

4. Добавить логирование одного структурированного объекта после расчета:
   - размеры изображения;
   - найденные landmarks и их `visibility` / `presence`;
   - `frontBounds`;
   - `scaleCmPerPx`;
   - выбранные `chestSpan`, `waistSpan`, `hipSpan`;
   - итоговые значения и методы расчета.

### Критерии готовности

- На проблемном фото видно, где алгоритм поставил грудь, талию и бедра.
- Можно отличить ошибку MediaPipe от ошибки маски и от ошибки выбора сечения.
- Пользователь видит не просто "недостаточно уверенно", а конкретную причину.

## Этап 2. Валидация входного фото

Цель этапа: не пытаться измерять заведомо неподходящие фото.

### Задачи

1. Использовать `visibility` и `presence` landmarks.
   Минимальные требования для фронтального фото:
   - оба плеча видны;
   - оба бедра видны;
   - колени или стопы видны для корректного масштаба по росту;
   - плечи и бедра находятся в ожидаемом порядке по Y;
   - человек занимает достаточную часть кадра.

2. Проверять фронтальность позы.
   Простые признаки:
   - горизонтальное расстояние между плечами достаточно большое;
   - горизонтальное расстояние между бедрами достаточно большое;
   - вертикальная асимметрия плеч и бедер не слишком велика;
   - центральная линия плеч и бедер не сильно смещается.

3. Проверять положение рук.
   Для автоматического измерения торса лучше требовать руки слегка в стороны.
   Если локти/запястья находятся близко к бокам торса, нужно:
   - предупреждать, что руки могут попасть в мерки;
   - либо не выдавать автоматическую грудь/талию;
   - либо использовать part-aware torso mask и понижать confidence при fallback.

4. Улучшить текст подсказок в модалке.
   Сейчас написано "руки вдоль тела или слегка в стороны". Для текущего алгоритма это вредно: руки вдоль тела часто сливаются с корпусом. Лучше: "руки слегка отведите от тела, чтобы между руками и корпусом был виден фон".

### Критерии готовности

- Фото с руками, прижатыми к корпусу, получает понятное предупреждение.
- Фото по пояс, фото сбоку вместо фронта, фото с сильным поворотом корпуса не принимаются как валидные.
- Ошибки валидации не выглядят как сбой программы.

## Этап 3. Переработка зон поиска мерок

Цель этапа: перестать выбирать плечевой пояс как грудь.

### Текущая проблема

Сейчас грудь ищется в диапазоне примерно `10-24%` от плеч до бедер и выбирается максимальная ширина. Это слишком высоко. На многих фото максимум в этой зоне будет соответствовать плечам, подмышкам или рукам.

### Новая логика

1. Определить torso coordinate system:
   - `shoulderY` = средний Y плеч;
   - `hipY` = средний Y бедер;
   - `torsoDelta = hipY - shoulderY`.

2. Задать анатомические окна:
   - грудь: примерно `0.25-0.42 * torsoDelta` ниже плеч;
   - талия: примерно `0.48-0.70 * torsoDelta` ниже плеч;
   - бедра: примерно `0.88-1.18 * torsoDelta` от плеч, с учетом actual hip landmarks.

3. Для груди не выбирать просто максимум.
   Лучше:
   - искать стабильное сечение ниже подмышек;
   - штрафовать слишком широкий span, близкий к ширине плеч;
   - штрафовать сечения, где рядом находятся локти/руки;
   - не разрешать `span.len` быть близким к полной ширине плечевого пояса.

4. Для талии выбирать локальный минимум только после груди и до бедер.
   Текущая идея с минимумом правильная, но ей нужен более чистый torso mask.

5. Для бедер использовать не только максимум рядом с hip landmarks.
   Нужно учитывать:
   - одежда может расширяться;
   - руки могут закрывать бок;
   - hip landmarks MediaPipe часто лежат внутри тела, а не на внешнем контуре.

### Критерии готовности

- На overlay линия груди не проходит через плечи/подмышки.
- При ошибочной зоне скан возвращает предупреждение, а не уверенное неверное число.
- Талия не выбирается сразу под грудью.

## Этап 4. Нормальная сегментация человека

Цель этапа: заменить цветовое отделение от фона на person segmentation.

### Почему текущая маска плохая

`buildForegroundMask` берет средний цвет по краям изображения и считает foreground всё, что отличается от этого цвета. Это ломается, если:

- фон неоднородный;
- есть тени;
- одежда похожа на фон;
- волосы или мебель попадают в компоненту человека;
- руки сливаются с торсом;
- фото сделано в комнате, а не на однотонном фоне.

### Варианты реализации

1. MediaPipe Image Segmenter в браузере.
   - Плюс: остается клиентская архитектура.
   - Плюс: не нужен backend.
   - Минус: дополнительная модель, загрузка и производительность.

2. BodyPix / TensorFlow.js segmentation.
   - Плюс: понятная body/person segmentation.
   - Минус: тяжелее, зависимости и CDN.

3. Серверная сегментация.
   - Плюс: стабильнее и контролируемее.
   - Минус: нужна инфраструктура, задержка, хранение фото, стоимость.

### Рекомендация

Для дипломного проекта и текущей архитектуры начать с клиентского MediaPipe Image Segmenter. Если производительность или качество не устроят, вынести сегментацию на backend позже.

### Задачи

1. Добавить отдельный слой `personMask`.
2. Нормализовать маску:
   - threshold по probability;
   - morphological close/open;
   - оставить компоненту, ближайшую к pose torso anchor.
3. Для торсовых мерок строить `torsoMask`, а не общий `personMask`.
4. В debug mode показывать маску полупрозрачным overlay.

### Критерии готовности

- На фото с неоднородным фоном человек выделяется лучше, чем текущим color threshold.
- Линии мерок строятся по телу, а не по мебели/теням/краям одежды.

## Этап 5. Исключение рук из торсовых мерок

Цель этапа: руки не должны увеличивать грудь, талию и бедра.

### Подход

1. По landmarks построить примерные capsules для рук:
   - shoulder -> elbow;
   - elbow -> wrist, если wrist доступен;
   - задать радиус capsule от масштаба тела.

2. Вычесть эти capsules из `personMask` при расчете торса.

3. Если после вычитания торс разрывается или сечение становится нестабильным:
   - снижать confidence;
   - просить другое фото;
   - переводить мерку в `statFallback`.

4. Если руки прижаты к телу и между рукой и корпусом нет фона:
   - не пытаться "угадывать" автоматически;
   - явно написать: "Руки сливаются с корпусом, автоматическая грудь/талия ненадежны".

### Критерии готовности

- Руки, опущенные вдоль тела, не расширяют линию груди/талии без предупреждения.
- На фото с руками слегка в стороны грудь считается стабильнее.

## Этап 6. Переработка расчета обхватов

Цель этапа: разделить измеренные значения и статистические догадки.

### Грудь, талия, бедра

1. Фронтальное фото дает ширину.
2. Профильное фото дает глубину.
3. Если профиль есть:
   - считать обхват как эллипс по ширине и глубине;
   - проверять согласованность фронта и профиля.
4. Если профиля нет:
   - использовать антропометрическую аппроксимацию;
   - помечать результат как "примерный";
   - снижать confidence.

### Бицепс

Текущий расчет через длину плечо-локоть нужно заменить.

Новая логика:

1. Найти середину upper arm: между shoulder и elbow.
2. Взять сечение руки перпендикулярно сегменту shoulder-elbow.
3. Измерить ширину по маске.
4. Если рука не отделена от торса, fallback на статистику с предупреждением.

### Нога

Текущий расчет через длину hip-knee нужно заменить.

Новая логика:

1. Найти верх бедра: область между hip и knee.
2. Взять несколько сечений перпендикулярно сегменту hip-knee.
3. Выбрать стабильную максимальную ширину верхней части бедра.
4. Если ноги слиты или одежда скрывает контур, fallback с предупреждением.

### Критерии готовности

- Методы расчета явно подписаны: `measured`, `frontApprox`, `statFallback`.
- Значение, полученное через fallback, не выглядит как уверенное измерение.
- Бицепс и нога перестают зависеть напрямую от длины конечности.

## Этап 7. UX проверки результата

Цель этапа: даже если автоматический скан остается приблизительным, пользователь должен видеть статус качества и иметь возможность проверить итоговые числа без ручного перетаскивания линий.

### Задачи

1. После расчета всегда показывать поля параметров, если есть хотя бы частичный результат.
2. Рядом с каждым параметром показывать статус:
   - "измерено";
   - "примерно";
   - "нужно проверить";
   - "не удалось".

3. Не блокировать применение параметров только из-за низкого confidence.
   Вместо этого показывать предупреждение и давать пользователю отредактировать значения.

### Критерии готовности

- "Недостаточно уверенно" не превращается в тупик.
- В демо всегда есть понятный путь к результату.

## Этап 8. Калибровка и тестовый набор

Цель этапа: перестать менять коэффициенты на глаз.

### Тестовый набор

Собрать локальный набор изображений, не коммитя приватные фото:

- фронт, хороший фон, руки слегка в стороны;
- фронт, руки вдоль тела;
- темная одежда на темном фоне;
- светлая одежда на светлом фоне;
- оверсайз;
- профиль;
- разные типы фигур;
- мужские и женские фото.

Для каждого теста хранить ожидаемый диапазон, а не одно точное число:

```json
{
  "heightCm": 175,
  "gender": "female",
  "expected": {
    "chest": [82, 98],
    "waist": [62, 82],
    "hips": [88, 108]
  }
}
```

### Метрики

1. Detection success rate.
2. Доля фото, где алгоритм корректно отказывается от автоматического измерения.
3. MAE по меркам на хороших фото.
4. Количество случаев, где грудь проходит через плечи.
5. Количество случаев, где руки попали в торсовые сечения.

### Автоматизация

На первом этапе достаточно тестового harness в браузере или node-compatible pure functions:

- вынести геометрию расчета в отдельный модуль;
- покрыть функции выбора сечений synthetic masks;
- сделать fixtures для landmarks и masks;
- не тестировать MediaPipe как внешнюю модель, а тестировать наш post-processing.

### Критерии готовности

- Изменения коэффициентов можно проверять на фиксированном наборе.
- Улучшение/регресс видно до ручного кликанья в UI.

## Этап 9. Возможный backend-вариант

Если клиентская сегментация окажется нестабильной, стоит вынести анализ на backend.

### Схема

1. Frontend отправляет фото на backend.
2. Backend запускает pose + segmentation модель.
3. Backend возвращает:
   - параметры;
   - confidence;
   - overlay/debug data;
   - причины предупреждений.

### Плюсы

- Версии моделей фиксируются.
- Можно добавить более тяжелые модели.
- Легче логировать реальные причины ошибок.
- Можно улучшать алгоритм без redeploy статики.

### Минусы

- Фото тела становятся backend data, нужны privacy decisions.
- Нужна инфраструктура и лимиты загрузки.
- Вырастает latency.
- Для дипломного демо это может быть избыточно.

## Рекомендуемая последовательность работ

### Итерация 1: быстрые улучшения без новой модели

1. Добавить debug overlay и вывод `qualityReasons`.
2. Использовать `visibility/presence` landmarks.
3. Изменить подсказку: руки слегка отвести от тела.
4. Перенести окно груди ниже и убрать выбор максимума в зоне плеч.
5. Разделить "измерено" и "примерно" в UI.

Ожидаемый эффект: меньше ложных результатов и понятнее причины отказов.

### Итерация 2: сегментация и вычитание рук

1. Подключить person segmentation.
2. Построить `torsoMask`.
3. Вычитать примерные masks рук.
4. Пересчитать грудь/талию/бедра по новой маске.

Ожидаемый эффект: заметно меньше случаев, где руки и фон портят мерки.

### Итерация 3: part-aware segmentation и тестовый набор

1. Усилить `torsoMask`: анатомический коридор торса плюс капсулы рук.
2. Вынести расчет геометрии в тестируемый модуль.
3. Собрать fixtures и регрессионные тесты.

Ожидаемый эффект: руки рядом с корпусом меньше расширяют автоматические сечения груди/талии.

### Итерация 4: профиль и точность

1. Сделать профильное фото не просто "опциональным", а рекомендуемым для точности.
2. Улучшить matching уровней фронт/профиль.
3. Явно показывать точность: без профиля результат приблизительный.

Ожидаемый эффект: меньше странных чисел из-за угаданной глубины тела.

## Следующий шаг: усиленная body-part segmentation

Реализуется без ручной коррекции линий. Вместо общей маски человека для торса строится part-aware torso mask:

1. Оставляем внешнюю person segmentation как источник силуэта.
2. Сначала пробуем semantic body-part segmentation через BodyPix `segmentPersonParts`.
3. Из BodyPix берем torso front/back и upper-leg front/back как torso+hips mask; руки, лицо, голени и стопы исключаются.
4. Если BodyPix недоступен или mask слишком слабая, по MediaPipe Pose строим анатомический коридор торса между плечами и бедрами.
5. В pose fallback руки вычитаются не простым порогом по цвету, а по капсулам shoulder-elbow-wrist с учетом того, где они пересекают коридор торса.
6. В diagnostics сохраняется источник `bodyPixParts` или `poseParts`, доля удаленных пикселей и причина fallback.
7. Синтетический тест должен проверять, что внешний body-part mask предпочитается pose fallback, а руки рядом с корпусом не расширяют талию/грудь относительно чистого силуэта.

Ожидаемый эффект: если руки или складки общей person mask расширяют горизонтальные сечения, расчет будет брать именно torso-part, а не самый широкий общий силуэт.

## Реализованный шаг: multi-mask fallback до полного геометрического результата

Цель: получить хотя бы один smoke-кейс, где `chest`, `waist`, `hips`, `arm`, `leg` рассчитаны без `statFallback`.

1. Грудь оставляем на строгой `bodyPixParts/poseParts` mask, потому что здесь главный риск - плечи и руки.
2. Если талия не найдена на строгой mask, запускаем `softPersonMask` fallback: person mask + анатомический коридор вокруг центра корпуса, без агрессивного вычитания рук.
3. Если бедра не найдены на строгой mask, запускаем отдельный `softPersonMask` fallback в окне бедер.
4. Значения, найденные через soft fallback, остаются геометрическими (`frontApprox` или `measured` при валидном профиле), но получают пониженный confidence и причину качества.
5. В diagnostics добавляется `softFallbackParts`, чтобы видеть, какие мерки были спасены мягкой mask.
6. Browser smoke должен явно находить кейсы, где `chest/waist/hips` не `statFallback`.

Результат реализации:

- В `analyzeBodyScan` добавлен `softPersonMask` fallback для талии и бедер. Он запускается только после провала строгой torso mask и ограничен анатомическим коридором вокруг линии плечи-бедра.
- Такие сечения остаются геометрическими, но получают пониженный confidence и причину `сечение найдено по мягкой маске силуэта`.
- В `diagnostics.softFallbackParts` видно, какие параметры были спасены мягкой маской.
- В `scripts/test_body_scan_core.mjs` добавлен synthetic regression test, где строгая torso mask содержит только верх торса, а талия/бедра должны добираться через `softPersonMask`.
- В `scripts/body_scan_browser_smoke.mjs` добавлен критерий `fullGeometricSuccess`: smoke падает, если ни одно фото не посчитало `chest`, `waist`, `hips`, `arm`, `leg` без `statFallback`.
- Локальный smoke нашел полный кейс `front-side-female-white-shirt`: `chest/waist/hips = measured`, `arm/leg = maskSection`. Талия и бедра при этом низкоуверенные и помечены как `softPersonMask`, поэтому UI честно предупреждает о качестве.

## Основные риски

- Полная автоматизация по одному фото останется нестабильной на произвольных пользовательских изображениях.
- Сегментация в браузере может быть тяжелой для слабых устройств.
- Оверсайз одежда скрывает реальные контуры тела; алгоритм будет измерять силуэт одежды, а не тело.
- Фото с руками вдоль тела принципиально плохо подходят для автоматического торса без хорошей body-part segmentation.
- Рост как масштаб работает только если человек виден полностью и mask bounds соответствуют реальному росту.

## Итоговая рекомендация

Не пытаться "починить" это одним порогом confidence. Правильная стратегия:

1. Сначала сделать ошибки наблюдаемыми через overlay и причины.
2. Затем исправить анатомические окна, особенно грудь.
3. Потом заменить цветовую маску на сегментацию человека.
4. После этого усилить body-part torso mask и добавить регрессионный тестовый набор.
5. В конце показывать причины качества и разрешать править итоговые численные значения перед применением.

Такой путь даст полезный результат в демо без обещания невозможной точности на любом фото.
